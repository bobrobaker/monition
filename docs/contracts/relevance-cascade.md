# Contract: relevance-cascade

Durable artifacts for the relevance-cascade filter (Phase 5). Owns interface semantics
for the label dataset, the trained head, and per-firing score logging. Buckets reference
the relevant section; they do not restate it.

Background: `docs/decisions/2026-06-18-noise-targets-the-filter-not-the-gate.md`,
*Update — 2026-06-21*. Spike: branch `spike/relevance-cascade`.

## 1. Label dataset (owned by B01)

A `(prompt, row) → label` table used to train and evaluate the relevance head.

- **location + format** (concretized B01): `data/relevance-cascade/labels.jsonl`
  (gitignored), **JSONL** — one record per line. Regenerated by
  `tools/build_relevance_labels.py`, which reads the hub through the single approved
  reader (`monition.export.export_records`), never the store directly. Rerunnable and
  deterministic (the split is a content hash, no RNG).
- **unit**: one *firing* — a (prompt context, takeaway row) pair that fired `on_demand`.
- **fields**:
  - `takeaway_id` — the row that fired (FK to `takeaways.id`).
  - `prompt` — the **full** user prompt, because that is what the runtime embeds at
    inference (`on_demand_match(prompt)` → `embed.semantic_scores`). Source = the
    firing's `situation` (= `prompt[:SITUATION_CHARS]`, ~4000) when present, else the
    lossy ≤200-char `trigger_context` preview. **Do NOT use `trigger_context` as the
    primary** — it is a ≤200 preview the runtime never embeds; training on it is a
    silent train/infer mismatch (B01 red-team finding C2, 2026-06-21).
  - `prompt_source` ∈ {`situation`, `trigger_context`} and `prompt_chars` — record which
    field fed each row so B02 can exclude/down-weight the lossy fallbacks.
  - `prompt_group` — `norm_prompt(prompt)` (lowercased, whitespace-collapsed, 120-char
    truncation — the spike's key). NOT the split axis; emitted so B02 can do
    leave-prompt-out or doubly-disjoint CV.
  - `label` ∈ {`helpful`, `noise`} — the rating, in ONE column. Human rows carry the
    human rating verbatim; oracle rows (a later volume-aid) map `oracle_relevant`
    {1,0} → {`helpful`, `noise`}. A single `label` column keyed by `label_source`
    beats parallel `label`/`oracle_relevant` columns (one schema, one read path).
  - `label_source` ∈ {`human`, `oracle`} — keeps the two distinguishable; never
    silently mixed, because the held-out test set must be **human-labeled only**.
  - `split` ∈ {`train`, `test`} — assigned by **row-disjoint grouping** (all firings of
    a `takeaway_id` go to the same split), NOT prompt grouping. The head embeds
    prompt⊕ROW over only ~46 distinct rows, so a *prompt*-grouped split leaks row
    identity: a prompt-ignoring per-row prior scored AUC ~0.77 on the prompt-grouped
    test ≈ the headline. Under the row-disjoint split that prior collapses to ~0.5
    (verified, asserted ≤0.6 in the builder), so any AUC the head earns is real
    prompt×row signal (B01 red-team finding C1, 2026-06-21).
  - **gate caveat for B02**: a *single* row-disjoint split is tiny and imbalanced at 46
    rows (~7 test rows, ~6 positives). The honest gate is **leave-row-out CV** (every
    row held out once → all firings get an unseen-row prediction, baseline still ~0.5),
    measured against the AUC **confidence interval**, not a point estimate. The `split`
    column is a final human-only sanity check, not the gate itself.
- **conservation**: every rated firing in the hub is either in the dataset or counted in
  a `skipped` tally with a reason (`not_on_demand`, `no_prompt_text`, etc.). Report the tally.
- **provenance**: regenerated from the hub, not hand-edited. Contains real session
  prompts → **never committed to git** (see spike `.gitignore`); lives under a gitignored
  data dir.

## 2. Trained head artifact (owned by B02)

The learned relevance head `L2′`, serialized for the runtime to load.

- **inputs at train/infer time**: L2-normalized prompt embedding ⊕ row embedding (the
  full vectors from `embed._embed_raw`, NOT cosine). Feature construction MUST match
  between training and inference — name it as an invariant. **Parity is about the input
  STRING, not just the embedding call** (B01 red-team C2): both halves are pinned —
  - *prompt half*: the **full prompt**. Train embeds `situation` (= `prompt[:4000]`);
    inference embeds the live `prompt`. B04 MUST truncate the runtime prompt to
    `SITUATION_CHARS` (or B02 must train on the same cap) so the two match; only 5/129
    prompts reach the cap, but pin it rather than leave it implicit.
  - *row half*: `f"{one_liner} {trigger_spec}"` — exactly as `on_demand_match` builds
    `texts` (`store_write.py`). B02 re-derives this from `takeaway_id` (the dataset does
    not store row text); `export.py` exposes `one_liner` but not `trigger_spec`, so read
    the takeaway row directly (`Store` exposes `Takeaway.trigger_spec`).
- **artifact**: weights + the exact feature spec (model name/dim, normalization, feature
  layout, any PCA basis) + a `version` + the train/test AUC at creation. Format and
  on-disk location are an **open decision** — left **unresolved (deferred)** because B02
  NO-GO'd and no production artifact was serialized
  (`docs/decisions/2026-06-21-relevance-cascade-b02-no-go.md`). The implemented serializer
  (`src/monition/relevance/head.py` — JSON: weights + feature spec + `version` + model id +
  AUCs, with model-id refusal on load) is the *candidate* format if the workstream resumes;
  on-disk location stays a versioned file near the managed cache `embed._weights_dir()`,
  NOT the Dolt store (weights are not row data).
- **embedding-version coupling**: the head is only valid for the embedding model it was
  trained against (`embed.MODEL_NAME`). Store that model id in the artifact; the runtime
  refuses to load a head whose model id ≠ the live `embed.MODEL_NAME`.
- **validity**: an artifact ships only if it clears the B02 usefulness gate — measured by
  **leave-row-out CV** on human labels (§1 gate caveat), against the AUC confidence
  interval, beating the ~0.5 row-disjoint baseline — not a single-split point estimate
  (operating-point chosen in B05, not at train time).

## 3. Per-firing score logging (decision in B01, wiring in B04, may touch firings schema)

To accumulate future training data, the runtime MAY log, per firing, the L2′ score and/or
the prompt embedding. This is an **addition to the firings contract** (`docs/contracts/
takeaway-store.md`) if pursued.

- **decision (B01, 2026-06-21)**: log the **scalar L2′ score + `head_version`** per firing
  — NOT the full embedding. The prompt is already stored (`situation` / `trigger_context`),
  so the embedding is always reconstructable on demand; one float + a version id is cheap,
  back-compat-safe, and the version makes the score interpretable across retrains
  (anti-substitution, below). Rejected: log-nothing (loses the cheap fire-time signal a
  future retrain wants); log-full-embedding (heavy row data, and re-embeddable anyway).
  Wiring is B04, not here.
- **if a firings column is added**: bump the firings contract, keep it nullable
  (back-compat), and remember Dolt omits NULL columns from JSON (`row.get`, not `row[]`).
- **anti-substitution**: a logged score is tied to the *head version* that produced it —
  log `head_version` alongside, or the score is uninterpretable after a retrain.
- **wired (B04, 2026-07-03 — schema v9)**: `firings.relevance_score decimal(6,5)` (the
  head's P(helpful) for this prompt×row pair) + `firings.head_version varchar(64)`
  (`"head-v{artifact.version}"`). Producer: `prompt_hook`'s disclosure only (the passive
  `on_demand` path through the cascade); both columns are NULL for pulls, other trigger
  kinds, cascade-disabled runs (`MONITION_CASCADE_DISABLE=1`), hosts without a staged
  artifact, and all pre-v9 rows. Cascade-suppressed candidates write **no firing row at
  all** (same rationale as the boilerplate gate: no row to batch-mis-rate); suppression
  counts go to the state log (`[cascade] suppressed N of M`). The read-side `Firing`
  reader deliberately does not select the columns yet — widening it would break reads on
  un-migrated stores; the consumer that needs scores (B05 / retraining) widens it then.
